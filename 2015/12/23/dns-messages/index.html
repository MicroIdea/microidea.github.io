关于DNS报文

Posted by Will on 2015-12-23

关于DNS报文的格式，备忘备忘。

域名协议中所有的通信都是由同一种格式的报文来承载的，简单好记，粗看分为5个区块：

  1. Header: 报文头部
  2. Question: 问题区
  3. Answer: 回答区
  4. Authority: 权威信息区
  5. Additional: 附加信息区

(RRs: Resource Records)

问题区包含: a query type (QTYPE), a query class (QCLASS), and a query domain name (QNAME)
剩下的3个区格式都一样: 一个可能为空的相连的资源记录列表(RRs)，各个区的信息上面的ascii表格都说明了

0x01 Header

一行两个字节，Header共12个字节

ID: 16个bit的标示符，是由发出DNS请求的程序生成的，每次请求都不一样，DNS服务器回答的时候这个标示符会原样copy，这样问题和回答就能一一对应了

QR: 只有1个bit，0表示该报文是一个问题(query)，1表示是一个回答(response)

OPCODE: 4个bit，用来表示这个报文的请求类型，当然是由问题的发起者设置了，回答的时候这个也是原样copy
0 a standard query (QUERY) 普普通通的请求
1 an inverse query (IQUERY) 逆向请求，知道ip问域名
2 a server status request (STATUS) 问问服务器的状态
3-15 留着将来说不定有用

AA: Authoritative Answer，1个bit，只在应答报文中有效，表示该回答是否由权威服务器发出

TC: TrunCation，1个bit，表示该报文是否因为大小超出传输层上限而分片了

RD: Recursion Desired，1个bit，由发问方设置DNS服务器原样copy，如果该bit置1则表示希望DNS服务器可以递归地执行请求（但是服务器并不一定支持递归，DNS查询方式可以是非递归、递归、迭代或者以上的组合）

RA: Recursion Available，1个bit，由DNS服务器设置，表示该服务器是否支持递归查询

Z: 1个bit，保留用，置0 （RFC1035中为3个bit）

AD: Authentic Data，1个bit，DNS服务器设置，表示answer和authority区的数据都是经过认证的（RFC2535新增）

CD: Checking Disabled，1个bit，发问方设置，表示是否接受非认证数据（RFC2535新增）

RCODE: Response code，4个bit，由DNS服务器设置
0 无错误
1 格式错误，服务器无法解析请求
2 服务器内部错误
3 域名错误，仅对权威域名服务器的响应有意义，表示该域名不存在
4 未实现，该请求的类型域名服务器尚不支持
5 拒绝，服务器傲娇
6-15 保留

QDCOUNT: 无符号16bit整型，表示问题区的条目数

ANCOUNT: 无符号16bit整型，表示回答区的资源记录数

NSCOUNT: 无符号16bit整型，表示权威信息区的资源记录数

ARCOUNT: 无符号16bit整型，表示附加信息区的资源记录数

以上就是DNS报文Header的所有信息

0x02 Question

问题区，包含QDCOUNT（通常都是1）条记录，每条记录的格式都是上图的样子

QNAME: 长度可变，由一个表示域名Label长度的字节后再加上Label组成的数据，结尾一个全0字节，因为root zone算是一个null label。举个例子：www.google.com表示为 3www6google3com0
这组数据不仅长度不定而且可以是奇数个字节，后面也不要填充（padding）

QTYPE: 2个字节，表示请求类型

QCLASS: 2个字节，表示请求的种类（Class），一般都设置成IN（for Internet）,还有其它种类但是一般情况不会用到

0x03 Resource Record

Question区下面的三个：Answer, Authority和Additional格式全部一样，见上图，都存储着资源记录（RRs），记录的数量在Header里面已经有说明了

NAME: 不定长，本条资源记录相关的1个域名

TYPE: 2个字节，表示RDATA区的资源类型，和QTYPE一致

CLASS: 2个字节，表示RDATA区的资源种类，和QCLASS一致

TTL: 32bit无符号整型，表示改资源记录可以缓存多少秒，0表示改记录只能用一次，不可缓存

RDLENGTH: 16bit无符号整型，表示RDATA区的字节数

RDATA: 资源记录本体，不定长，格式由TYPE和CLASS决定，e.g. 如果TYPE为A，CLASS为IN，那么RDATA区就是4个字节的ARPA Internet address（就是普通的IPv4地址…）

0x04 信息压缩

为了减小报文长度，当多条域名含有相同部分的时候可以使用指针来压缩信息，这样一个报文中的域名就有3种表示方法：

  1. 一序列的label加上一个全0的字节作结尾（普通向）
  2. 只有1个指针（简洁向）
  3. 一序列的label加上一个指针作结尾（混合向）

怎么区分数据是一个普通的label还是一个特殊的指针？

因为一个label规定是必须小于等于63个字符的，所以label前表示长度的一个字节开头两个bit必然是0，而指针则开头两个bit全部为1，画个图：

因为指针是为了压缩域名中的重复部分，所以肯定是指向header之后包含域名的数据了，再举个例子：

报文中含有www.google.com和images.google.com，重复部分是google.com，信息压缩后就是（用ascii表示）

3www6google3com0（假设其中6这个字节在报文中的偏移量是20）和6images19220

Answer: 里面有两条记录，第一个就是我们直接键入的网址www.bing.com，但是这条记录是个CNAME，就是一个别名，指向了cn.a-0001.a-msedge.net，这是因为本人在中国地区且没有挂代理，如果是在其他国家就会返回不同的记录了。这也是大部分CDN的原理，选择离用户最近的服务器，提高响应速度，不同语言地区显示不同的页面。
第二条是个普通的A记录，就是cn.a-0001.a-msedge.net的IPv4地址

Authority: 权威信息区里面也有两条记录，type都是NS（Name Server），表示www.bing.com这个域名的其实是托管在ns2.a-msedge.net和ns1.a-msedge.net这两台域名服务器的，这两个服务器就是它的权威or授权服务器
P.S. 本人当时用了阿里的DNS（223.5.5.5）,后换用bing的权威域名服务器204.79.197.1，之后返回的信息只有Answer和Authority，就没有Additional了

Additional: 附加信息区也有两条记录，type A，就是上述两台权威服务器的IPv4地址了
